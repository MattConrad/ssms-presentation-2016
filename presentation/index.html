SQL Server Management Studio (SSMS): used for database administration and ad-hoc interaction with your database.


Programmers typically use SSMS for testing queries, but there's a lot more.


(Another common way to interact with the db ad-hocly is to use the built in database access tools with Visual Studio. These work similarly to SSMS, and much of tonight's presentation applies to those tools as well.

Getting SSMS



If you already have a database engine installed (like LocalDB/SQL Server Express that comes with Visual Studio), you can get SSMS standalone at https://msdn.microsoft.com/en-us/library/mt238290.aspx.


Or you can download SQL Server Developer edition for free at https://www.microsoft.com/en-us/sql-server/sql-server-editions-developers.

A quick remark about tonight's presentation:


Like Visual Studio, SSMS is really big. I will be skipping over things that I think are obscure, less important, or that I don't use/don't understand very well. If we pass over some items without remark, that's why. (Probably.)


One significant omission worth mentioning: SQL Server Agent. SQL Agent lets you schedule SQL commands to run later, or on a repeated schedule. You’ll probably schedule backups using SQL Agent, and it tends to accumulate other responsibilities as your system grows. We don't have time to get into Agent tonight, but if you work with SQL Server much, you'll want to get familiar with it.

The Query Window


  • Enter a query and hit F5 (or Ctrl-E) to run it.
  • Open new query windows (each with a new active database connection) using Alt-N.
  • Comments are double dashes. Use Ctrl-K, Ctrl-C as a block comment shortcut
  • If you highlight part of the text and hit Run, only the highlighted text is run. This is very handy.
  • You can change your results format from grid, to text, to file output.

Browsing the Object Explorer tree


First, server level properties:


  • General: potentially useful stuff you might want to look up
  • Server/Memory: straightforward
  • Security: auth mode, very significant. Note the difference between database and server login info. Also, “sa” user.
  • Database settings: database default locations
  • Permissions: logins can get server level permissions

Databases folder: a quick stop and we'll return later.


SQL Server has the concept of system databases and user databases


System databases are built in databases used for managing the server itself


User databases are made by programmers to hold customized data. If you're not a DBA, you use user databases.



There's much more to talk about here. We'll come back to this in a while.

Security folder:


  • Logins: these are server level accounts. Database level users are related, but something different.
  • Server Roles: group permissions inside SQL Server, you don't have to apply permissions to every user individually

Replication:


Replication copies data from one SQL server to another, to keep data from multiple servers in sync. It's a whole topic of its own.


Management:


  • SQL Server Logs: Here’s one you’ll probably interact with. If you’re having weird server level issues (SQL Server won’t start, system databases are behaving strangely, etc), you can maybe get a clue from the server logs. Note that if you can’t even connect to SQL Server you can still view the logs from the filesystem, hit Google for “SQL Server log location” for how to find the log files for your SQL Server version.
  • On a full install of SQL Server, there will be more here, like database maintenance plans.
